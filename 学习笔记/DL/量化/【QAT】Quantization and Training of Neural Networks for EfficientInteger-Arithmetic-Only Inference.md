# 1. 核心量化方案
$$r = S*(q - Z)$$
- r 是原始模型浮点参数
- q 是量化模型的整型参数
- S 是量化的缩放系数
- Z 是量化的零点

## 1.1. 量化流程推导

考虑两个 $N * N$ 方阵 $r_1, r_2$ 的矩阵乘法，其乘积为 $r_3 = r_1 * r_2$ ，
把这三个矩阵 $r_\alpha(\alpha是1，2，3)$ 中的元素表示为 $r_\alpha^{(i,j)}, for\ 0 < {i, j} <= N$ ；每个矩阵对应的量化参数表示为 $(S_\alpha, Z_\alpha)$ ，量化之后的元素表示为 $q_\alpha^{(i, j)}$ ，则有如下公式：
$$r_\alpha^{(i, j)} = S_\alpha * (q_\alpha^{(i, j)} - Z_\alpha) \tag{2}$$ 根据矩阵乘法，有如下式子：
$$S_3*(q_3^{(i, k)} - Z_3) = \sum_{j=1}^N{S_1*(q_1^{i,j} - Z_1) * S_2*(q_2^{(j, k)} - Z_2)} \tag{3}$$ 由此可以整理得出量化之后的输出：
$$q_3^{(i, k)} = Z_3 + M\sum_{j=1}^N{(q_1^{(i,j)}-Z_1)(q_2^{(j, k)}-Z_2)} \tag{4}$$ 其中 M 的定义如下：
$$M:= \frac{S_1S_2}{S_3} \tag{5}$$
## 1.2. scale参数的整数表示方法

在公式(4)中，只有 M 不是整数。但是 M 是一个由 $S_1,  S_2, S_3$ 确定的常数。作者经验性的发现，M 是一个（0, 1）之间的小数，因此可以把它表示为规范化的形式：
$$M = 2^{-n}M_0 \tag{6}$$
其中，**$M_0$是$[0.5, 1)$ 之间的数， n 是非负整数（根据硬件支持情况，n通常为31）** 。
**公式(6)是怎么得到的？** $M = 2^{-n}M_0 \Longleftrightarrow 2^nM = M_0$  对于$(0,1)$之间的数，总能找到一个对应的 $n$ ，根据公式映射到 $[0.5,1)$ 中，比如：
$M = 0.1,n = 3 \Rightarrow 0.1 * 2^3 = 0.8$ 
$M = 0.4, n = 1 \Rightarrow 0.4 * 2^1 = 0.8$ 
$M = 0.6, n = 0 \Rightarrow 0.6 * 2^0 = 0.6$
公式(6)是一个很重要的变形，有了这个变形，才能保证后续的变换不会引入很大的误差。接下来需要把 $M_0$ 用定点整数表示。这里以int32表示$M_0$ 为例，取最接近 $2^{31}M_0$ 的整数来表示 $M_0$ (Q31编码):
$$int\_{M_0} = round(2^{31}M_0) \tag{7}$$
**注意** ，公式(7)重点是表示方法的变换，而不是数值的计算。这里考虑的是如何用int32整数表示浮点数。此公式计算出的 $int\_M_0$ 是一个很大的数，而这里的变大只是为了编码，在整数乘法之后，还会把 $2_{31}$ 除掉。根据公式（7），我们可以得出 $M_0$ 的近似表示方法：
$$M_0 \approx \frac{int\_M_0}{2^{31}} \tag{8}$$

| $M_0$实际值 | $M_0$存储值                       |
| -------- | ------------------------------ |
| 0.75     | $\approx 0.75 * 2^{31}$        |
| 0.5      | $\approx 0.5* 2^{31} = 2^{30}$ |
假设在int8推理时，有一个整数输入 $x$ ，实际计算过程为：
1. real scale计算：$M = \frac{s_{in}s_{w}}{s_{out}}$ 
2. normalize：$M = 2^{-n}M_0$，求出n
3. 计算出$int\_{M_0}$ 
4. 计算：$x * \frac{int\_M_0}{2^{31}} * 2^{-n}$ 
代码的计算形式如下：
```C++
(x * int_M0) >> (31 + n)
```
### 1.2.1. 为什么选Q31，而不是16或者8
要确保精度最大化，int32有32 bits，最高位是符号位，剩下31位表示小数，精度是：
$$\frac{1}{2^{31}} \approx 4.6*10^{-10}$$
### 1.2.2. 为什么要转换到 $[0.5, 1)$ 范围内
如果不限制 $M_0 \in [0.5,1)$ ，假设$M_0 = 0.01$ ，那么
$$2^{31} * 0.01 \approx 2^{25}$$
有效位只有25bits，精度下降很多。
当$M_0 \in [0.5,1)$ 时，$2^{31} * M_0 \in [2^{30}, 2^{31})$ , 高位都被占满，有效精度$>=30 bits$ 
用 **Q31 定点编码** 把一个 $[0.5,1)$  的浮点数，转换成一个**高精度整数近似值**，从而可以用整数乘法替代浮点乘法
### 1.2.3. 论文中强调要“round-to-nearest”
右移操作默认是向下取整，但是论文强调要round-to-nearest（四舍五入），否则：
1. 会引入系统误差（bias)
2. 神经网络会累积误差

### 1.2.4. 用Q31定点编码是因为要把模型量化到int32，还是说不管模型量化到什么位宽，都用Q31编码？int8量化时，也用Q31吗？

#### 1.2.4.1. Q31 定点编码与模型量化位宽的关系
##### 1. 核心结论
- **Q31 并不是因为模型量化到 int32**
- **Q31 是用于表示 scale（缩放因子）的高精度定点格式**
- **无论模型是 int8 / int16 / 其他量化位宽，通常都可以使用 Q31 来表示 scale**
##### 2. int8 量化时是否使用 Q31？
**是的，int8 量化通常也会使用 Q31 表示 scale**
典型 INT8 推理流程如下：

| 类型 | 位宽 |
|------|------|
| activation / weight | int8 |
| accumulator | int32 |
| multiplier（scale） | **int32（Q31）** |

也就是说：

> 即使模型是 int8，scale 依然用 int32（Q31）来表示

##### 3. 为什么不用 Q7 / Q15，而选择 Q31？
1. **精度对比**

| 格式 | 精度 | 说明 |
|------|------|------|
| Q7   | ≈ 1/2⁷ ≈ 0.0078 | 误差较大（约 1%） |
| Q15  | ≈ 1/2¹⁵ | 一般 |
| **Q31** | ≈ 1/2³¹ | 精度极高 |

2. **为什么 scale 需要高精度？**
量化推理中的关键公式：
$$y = \frac{s_x * s_w}{s_y} * \sum(x_i * w_i)$$
其中：
- $\sum(x_i * w_i)$ 是 int32
- scale 是浮点数
如果 scale 精度不足：
- 会产生系统性误差（bias）
- 多层网络中误差会累积
- 最终导致精度明显下降

##### 4. 为什么 Q31 “刚好合适”？

1. **计算流程匹配**

	```cpp
	int64_t tmp = (int64_t)x * int_M0;
	y = tmp >> 31;
	```

	- 使用 int64 避免溢出
	- 右移 31 位恢复 scale
	👉 这就是经典的 **31-bit fractional multiply**

2. **精度与硬件的平衡** 
	- int32 × int32 → int64（安全）
	- 31 bit 小数精度接近 float
	- 计算仍然高效（整数运算）
##### 5. 是否所有平台都使用 Q31？
1. **主流推理框架（TFLite / QNNPACK / ONNX Runtime）**
	- 使用 Q31 表示 scale
	- 标准做法
2. **低端 MCU（如 Cortex-M）**
	- 可能使用 Q15（int16）
	- 用精度换性能
3. **NPU / 专用加速器（如 Qualcomm Hexagon）** 
	- 不直接使用 Q31
	- 使用：
	    - fused scale
	    - per-channel multiplier + shift
	    - 或硬件专用格式
##### 6. 一句话总结
> **Q31 是“scale 的表示方式”，而不是“模型量化位宽”的选择**
##### 7. 工程视角总结
量化推理的实际计算路径：
```cpp
int8 × int8 → int32 accumulate
           ↓
   × (Q31 multiplier)
           ↓
        >> shift
           ↓
         int8
```

##### 8. 常见误区

❌ 错误理解：

> int8 量化 → 所有数据都应该是 int8

✅ 正确理解：

> 数据可以是低精度（int8），但 scale 必须高精度（Q31），否则误差不可控

### 1.2.5. Q31编码格式是软件参数量化表示，而不会在特定的硬件加速芯片上执行。

## 1.3. 零点的高效处理
这一段其实是在讲**量化矩阵乘法中 zero-point（零点）带来的额外计算，如何被“重写 + 折叠”掉**，从而把计算复杂度恢复到**普通整数矩阵乘法**。

我给你分三层讲清楚：**公式在干嘛 → 为什么要这么做 → zero-point 为什么必须特殊处理**

---

### 1️⃣ 这段公式在做什么？

从原始公式（你前面那张图）：

(q1−Z1)(q2−Z2)(q_1 - Z_1)(q_2 - Z_2)(q1​−Z1​)(q2​−Z2​)

展开：

=q1q2−Z1q2−Z2q1+Z1Z2= q_1 q_2 - Z_1 q_2 - Z_2 q_1 + Z_1 Z_2=q1​q2​−Z1​q2​−Z2​q1​+Z1​Z2​

---

#### 👉 论文的 (7) 式就是把它“整理成可预计算形式”

q3(i,k)=Z3+M(NZ1Z2−Z1a2(k)−Z2aˉ1(i)+∑q1q2)q_3^{(i,k)} = Z_3 + M \Big( N Z_1 Z_2 - Z_1 a_2^{(k)} - Z_2 \bar{a}_1^{(i)} + \sum q_1 q_2 \Big)q3(i,k)​=Z3​+M(NZ1​Z2​−Z1​a2(k)​−Z2​aˉ1(i)​+∑q1​q2​)

其中：

- a2(k)=∑jq2(j,k)a_2^{(k)} = \sum_j q_2^{(j,k)}a2(k)​=∑j​q2(j,k)​（列和）
- aˉ1(i)=∑jq1(i,j)\bar{a}_1^{(i)} = \sum_j q_1^{(i,j)}aˉ1(i)​=∑j​q1(i,j)​（行和）

---

### 2️⃣ 这一步的核心意义

👉 把原来每次都要做的：

(q1 - Z1)  
(q2 - Z2)

👉 变成：

##### ✔ 只做一次的东西（O(N²)）：

- 行和（aˉ1(i)\bar{a}_1^{(i)}aˉ1(i)​）
- 列和（a2(k)a_2^{(k)}a2(k)​）

---

##### ✔ 核心计算只剩：

∑q1q2\sum q_1 q_2∑q1​q2​

👉 就是**普通整数矩阵乘法**

---

### 3️⃣ 为什么论文说复杂度变成 O(N3)O(N^3)O(N3) 且“干净”？

原始 naive：

- 每个乘法：
    - 2 次减法
    - 1 次乘法

👉 很“脏”（有 zero-point）

---

现在：

|部分|复杂度|
|---|---|
|行和 / 列和|2N22N^22N2|
|核心乘法|N3N^3N3|

👉 所以：

> “everything else is O(N²)，核心是整数 GEMM”

---

### 4️⃣ 零点为什么需要特殊处理？（关键）

这是这段话最本质的问题。

---

#### 4.1 zero-point 的物理意义

量化公式：

xreal=s(xq−z)x_{\text{real}} = s (x_q - z)xreal​=s(xq​−z)

👉 zero-point 的作用：

> **让 0 能被整数精确表示**

例如：

- int8 范围：[-128,127]
- 但数据可能是非对称的

---

#### 4.2 问题来了：zero-point 会破坏乘法结构

理想情况（无 zero-point）：

xreal=sxqx_{\text{real}} = s x_qxreal​=sxq​

那么：

y=sxsw∑xqwqy = s_x s_w \sum x_q w_qy=sx​sw​∑xq​wq​

👉 非常干净，直接 GEMM

---

#### 4.3 有 zero-point 时：

(xq−zx)(wq−zw)(x_q - z_x)(w_q - z_w)(xq​−zx​)(wq​−zw​)

展开：

=xqwq−zxwq−zwxq+zxzw= x_q w_q - z_x w_q - z_w x_q + z_x z_w=xq​wq​−zx​wq​−zw​xq​+zx​zw​

👉 出现了：

- **额外的 cross term**
- 破坏了纯 GEMM 结构

---

### 5️⃣ 如果不处理 zero-point 会怎样？

👉 回到 naive：

每次乘法都要：  
(q1 - Z1)  
(q2 - Z2)

👉 带来：

- 2N32N^32N3 次减法
- cache 不友好
- SIMD / NPU 难优化

---

### 6️⃣ 这篇论文做的事情（本质）

👉 **把 zero-point 的影响“提取出来”，变成低阶项**

具体就是：

|项|处理方式|
|---|---|
|q1q2q_1 q_2q1​q2​|保留 → GEMM|
|−Z1q2-Z_1 q_2−Z1​q2​|→ 列和|
|−Z2q1-Z_2 q_1−Z2​q1​|→ 行和|
|Z1Z2Z_1 Z_2Z1​Z2​|→ 常数|

---

👉 结果：

> **zero-point 不再出现在 inner loop（N³）里**

---

### 7️⃣ 为什么论文说“low-overhead handling of arbitrary zero-points”？

👉 因为：

- zero-point 可以是任意值（非对称量化）
- 但通过这种展开：

👉 额外开销只剩：

O(N2)O(N^2)O(N2)

---

### 8️⃣ 一句话总结（最关键）

👉 **zero-point 会破坏“纯乘加结构”，必须单独拆出来，否则计算会膨胀到 O(N3)O(N^3)O(N3) 的额外开销。**

---

### 9️⃣ 给你一个工程直觉（很重要）

在实际框架（TFLite / QNN / TensorRT）里：

👉 都会做这件事：

- **zero-point folding**
- **bias correction**
- **precompute row/col sum**

👉 目的只有一个：

> **让 kernel 只做：int8 × int8 → int32 accumulate**

## 1.4. 为什么零点处理要用展开的形式，而不是先在矩阵中减去零点
你的这个想法在**数学上完全正确**，而且在“纸面复杂度”上确实是 2N22N^22N2 次减法 + N3N^3N3 次乘法，看起来和论文优化后的复杂度是同一个量级。

但论文不这么做，**核心原因不是算术复杂度，而是工程实现代价（memory / bandwidth / layout / fusion）**。下面给你拆到工程层面。

### 1️⃣ 你的方法 vs 论文方法（本质区别）

#### ✔ 你的方法（预减 zero-point）

A' = q1 - Z1  
B' = q2 - Z2  
C = A' × B'

特点：

- 需要生成 **两个新矩阵 A', B'**
- 每个元素减一次 → 2N22N^22N2
- 再做标准 GEMM

#### ✔ 论文方法（展开 + 折叠）

C = GEMM(q1, q2)  
  + row_sum / col_sum 修正  
  + 常数项

特点：

- **不改动原始矩阵**
- zero-point 被拆成低阶项
- GEMM 保持“原始数据布局”

### 2️⃣ 为什么论文不采用“先减再乘”？

#### 🚨 关键原因 1：内存带宽（最重要）

你的方法：

- 需要写出 A'（N²）
- 需要写出 B'（N²）

👉 多了：

2N2 次 memory write2N^2 \text{ 次 memory write}2N2 次 memory write

而 GEMM 本身是：

> **memory bandwidth-bound（尤其在移动端/NPU）**

👉 结论：

> **多一次 full tensor 写入，代价远大于少做一些算术**

#### 🚨 关键原因 2：cache locality 破坏

GEMM 的优化依赖：

- blocking / tiling
- cache reuse

如果你预处理：

load q1 → compute → write A'

👉 会导致：

- 数据被写回内存
- 再读入 GEMM

👉 cache reuse 被破坏

#### 🚨 关键原因 3：不利于 kernel fusion

现代推理框架（你应该很熟）：

- Conv + Bias + ReLU + Quant → **fused kernel**

如果你先做：

q1 → subtract Z1 → A'

👉 就变成：

- 一个独立 kernel
- 中间 tensor 需要 materialize

👉 破坏：

- fusion
- pipeline

#### 🚨 关键原因 4：NPU / SIMD 不支持这种模式

以 Qualcomm / ARM NEON 为例：

👉 硬件支持的是：

int8 × int8 → int32 accumulate

但不支持：

( int8 - const ) × ( int8 - const )

👉 如果你预减：

- 需要额外 kernel
- 或者增加指令开销

#### 🚨 关键原因 5：per-channel zero-point 更复杂

实际情况：

- Z1Z_1Z1​：可能是 per-tensor
- Z2Z_2Z2​：可能是 per-channel

如果你预减：

👉 就会变成：

B'[j,k] = q2[j,k] - Z2[k]

👉 每列不同 → 很难 vectorize

而论文方法：

👉 把它变成：

-Z1 * col_sum  
-Z2 * row_sum

👉 更规整

### 3️⃣ 为什么论文方法更“工程友好”？

论文方法本质是：

> **把 zero-point 从“数据路径”移到“标量路径”**

#### 数据路径（N³）

Σ q1 * q2

👉 完全标准 GEMM  
👉 可用：

- SIMD
- Tensor Core
- NPU

#### 标量路径（N²）

row_sum / col_sum

👉 轻量计算  
👉 不影响主 kernel

### 4️⃣ 一个非常关键的性能事实

在实际系统中：

> **一次额外的 memory pass ≈ 上亿次整数运算**

👉 所以：

- 减少 memory > 减少算术

### 5️⃣ 你方法什么时候是可行的？

其实不是完全不能用：

#### ✅ 情况 1：小矩阵（N 很小）

- cache 能装下
- memory 不敏感

#### ✅ 情况 2：离线预处理（权重）

对于权重：

w' = w - Z_w

👉 **是可以的（而且常用）**

因为：

- 权重是常量
- 只处理一次

#### ❌ 情况 3：activation（运行时）

👉 不适合：

- 每次 inference 都要做
- 成本太高

### 6️⃣ 一句话总结（最核心）

👉 **你的方法在“计算量”上没问题，但在“内存和硬件执行模型”上是低效的。**

### 7️⃣ 再给你一个更底层总结（很关键）

两种方法的本质区别：

|方法|zero-point 位置|
|---|---|
|你的方法|数据路径（tensor-level）|
|论文方法|标量路径（reduction-level）|


👉 而现代加速器的原则是：

> **让 tensor path 尽可能“干净”（纯乘加），把复杂性挪到标量路径**
