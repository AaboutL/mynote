# 高通8397 SoC的NPU芯片方案，然后判断Q31是否合适
## ✅ 结论（直接回答你核心问题）

👉 在高通（包括你说的“8397”这一类 Snapdragon SoC）NPU 上：

> **Q31 是“软件侧实现 scale 的通用方案”，但在 NPU 上通常“不是最终执行格式”**

更具体：

- ✅ **在 CPU / reference kernel / fallback path → Q31 非常合适（标准做法）**
- ⚠️ **在 Hexagon NPU（HTP）上 → 通常不会直接用 Q31**
- ✅ **NPU 更倾向于：INT8 / INT16 / mixed precision + 内部专用 scaling 机制**

## 1️⃣ 先澄清：8397 SoC 的本质架构

虽然“8397”不是公开常用 marketing 名称（更像内部编号），但可以确定属于：

👉 **Snapdragon + Hexagon NPU（HTP）架构**

核心结构：

- NPU = **Hexagon DSP / Tensor Processor**
- 统一调度框架 = **QNN / AI Engine**

并且关键能力：

- 支持 **INT4 / INT8 / INT16 / FP16 等多精度**
- 主流推理：**INT8 为核心路径**

## 2️⃣ Qualcomm NPU 的“真实计算模式”

这是判断 Q31 是否合适的关键。

### 2.1 NPU 并不是“通用整数 ALU”

NPU 本质是：

> **systolic array + tensor MAC pipeline**

特点：

- 专门做：
    
    int8 × int8 → accumulate (int32 / wider)
    
- scale / requantization：  
    👉 **通常融合在硬件 kernel 里**

---

### 2.2 Qualcomm 官方量化支持

从官方文档可以明确：

- 权重：int8
- 激活：int8 / int16

👉 没有提 Q31 这种“显式 multiplier”

## 3️⃣ Q31 在 Qualcomm 体系里的位置

这里是关键 insight（很多人搞错的点）：

---

### ✅ Q31 是“软件量化参数表示”

通常发生在：

- TFLite reference kernel
- gemmlowp
- CPU fallback
- 某些 DSP scalar path

👉 用于实现：

y=(x⋅int_M0)>>shifty = (x \cdot \text{int\_M0}) >> shifty=(x⋅int_M0)>>shift

---

### ❌ 但在 NPU 上：

👉 **不会真的执行这套 Q31 乘法**

原因有三个：

---

## 4️⃣ 为什么 Q31 不适合直接在 NPU 上用？

### 4.1 NPU 的数据通路不匹配

NPU pipeline 是：

int8 × int8 → int32 accumulate  
            → requantize (fused)

而 Q31 需要：

int32 × int32 → int64 → shift

👉 问题：

- 需要 64-bit intermediate
- 不符合 tensor core datapath

---

### 4.2 scale 通常被“融合掉”

在 QNN / Hexagon 上：

👉 scale 会被编译器处理为：

- per-channel scale
- fused multiplier + shift
- 或 LUT / microcode

👉 而不是显式 Q31 乘法

---

### 4.3 硬件有自己的定点格式

Hexagon NPU 内部更接近：

- Q8 / Q16 accumulators
- 或 block floating / mixed precision

👉 而不是 Q31 这种“软件定义格式”

---

## 5️⃣ 一个非常关键的现实证据（工程层）

来自实际开发经验（以及社区验证）：

> 不同 Snapdragon 芯片 INT8 精度表现不同，原因之一是  
> **NPU 内部 rounding / scaling 实现不一致**

👉 这说明：

- scale 处理是 **硬件实现**
- 不是统一的 Q31 算法

---

## 6️⃣ 那 Q31 还有没有意义？

👉 有，而且很重要：

---

### ✅ 场景 1：模型导出阶段

float scale → (multiplier, shift)

👉 这里通常用 Q31 计算

---

### ✅ 场景 2：CPU fallback

当 op 不支持 NPU：

👉 会走：

int32 × Q31 → shift

---

### ✅ 场景 3：跨平台一致性

你做推理框架时：

👉 Q31 是：

- 可移植
- 精度高
- deterministic

# 帮我把 QNN graph 里 scale 是怎么被“折叠进 kernel”的具体过程拆开（含 per-channel / bias correction / shift 分解）

- xq,wq,yqx_q, w_q, y_qxq​,wq​,yq​：量化后的 int8

- zx,zw,zyz_x, z_w, z_yzx​,zw​,zy​：zero-point（通常权重对称量化 zw=0z_w=0zw​=0）

卷积/全连接的真实值：

## 1️⃣ 编译期：确定输出缩放（per-channel）

QNN 默认是 **per-channel weight scale**（卷积每个输出通道一个 sw(c)s_w^{(c)}sw(c)​）：

real_multiplier(c)=sx⋅sw(c)sy\text{real\_multiplier}^{(c)} = \frac{s_x \cdot s_w^{(c)}}{s_y}real_multiplier(c)=sy​sx​⋅sw(c)​​

👉 这是**每个输出通道一个 multiplier**。

---

## 2️⃣ 编译期：bias 的“预折叠”（bias correction）

先把 bias 变成 **int32 域**，避免运行时浮点：

bq(c)=round(breal(c)sx⋅sw(c))b_q^{(c)} = \text{round}\left(\frac{b_{\text{real}}^{(c)}}{s_x \cdot s_w^{(c)}}\right)bq(c)​=round(sx​⋅sw(c)​breal(c)​​)

但还不够，因为存在 zxz_xzx​：

展开求和项：

∑i(xq−zx)wq=∑ixqwq−zx∑iwq\sum_i (x_q - z_x) w_q = \sum_i x_q w_q - z_x \sum_i w_qi∑​(xq​−zx​)wq​=i∑​xq​wq​−zx​i∑​wq​

👉 第二项是常数（只和权重有关），可**提前算好并折进 bias**：

b~q(c)=bq(c)−zx⋅∑iwq(c)\tilde{b}_q^{(c)} = b_q^{(c)} - z_x \cdot \sum_i w_q^{(c)}b~q(c)​=bq(c)​−zx​⋅i∑​wq(c)​

这一步就是你说的 **bias correction / folding**。

---

## 3️⃣ 编译期：multiplier 分解（shift + integer）

对每个通道：

$$real_multiplier(c)∈(0,1)\text{real\_multiplier}^{(c)} \in (0,1)real_multiplier(c)∈(0,1)$$

做标准规格化：

real_multiplier(c)=M0(c)⋅2−n(c),M0(c)∈[0.5,1)\text{real\_multiplier}^{(c)} = M_0^{(c)} \cdot 2^{-n^{(c)}},\quad M_0^{(c)} \in [0.5,1)real_multiplier(c)=M0(c)​⋅2−n(c),M0(c)​∈[0.5,1)

然后量化成整数：

multq(c)=round(M0(c)⋅231)\text{mult}_q^{(c)} = \text{round}(M_0^{(c)} \cdot 2^{31})multq(c)​=round(M0(c)​⋅231)

👉 得到：

- `mult_q[c]`（int32，Q31）
- `shift[c] = n^{(c)}`

⚠️ **关键点**：  
在 QNN/HTP 上，这一步**不一定以 Q31 形式保留**，而是会被编译器**重编码成硬件友好的 `(mul, shift)` 或更底层 micro-op**（下面解释）。

---

## 4️⃣ 编译期：进一步“硬件化”分解（HTP 关键）

Hexagon HTP 不直接执行：

int64 = acc * mult_q >> (31 + shift)

而是把它拆成更贴合 datapath 的形式，例如：

### 4.1 变成“高位乘法 + 可变右移”

acc32 → mul_hi(acc32, k[c]) → shift_right(s[c])

其中：

- `k[c]`：通常压缩到 **int16 / int24 有效位**
- `mul_hi`：只取高位（避免 64-bit 完整乘法）

👉 等价于近似 Q31，但更适合硬件 pipeline。

---

### 4.2 shift 拆分（两段式）

有些实现会把：

>>(31+n)>> (31 + n)>>(31+n)

拆成：

- 固定 shift（如 15 或 16，匹配乘法单元）
- - 可变 shift（per-channel）

👉 减少控制复杂度

---

### 4.3 per-channel 向量化

所有这些参数最终变成：

mult[c], shift[c], bias[c]

并以 **vector 常量**形式喂给 kernel。

---

## 5️⃣ 运行期：HTP kernel 实际执行

最终 kernel（逻辑上）做的是：

// 1. int8 MAC  
acc[c] = Σ (x_q * w_q)         // int32  
  
// 2. 加偏置（已fold）  
acc[c] += bias_q[c]  
  
// 3. requant（已硬件化）  
acc[c] = MUL_AND_SHIFT(acc[c], mult[c], shift[c])  
  
// 4. 加输出 zero-point  
y_q[c] = acc[c] + z_y  
  
// 5. clamp 到 int8

👉 注意：

- **没有显式 Q31**
- 但数学上等价于 Q31 + shift

---

## 6️⃣ rounding 的处理（很关键）

HTP 通常做的是：

- **round to nearest**
- 有时是 **round to nearest even**

实现方式类似：

(acc + (1 << (shift-1))) >> shift

或在 `mul_hi` 阶段内嵌 rounding

---

## 7️⃣ 为什么必须 per-channel？

如果不用 per-channel：

sw(c) 差异很大s_w^{(c)} \text{ 差异很大}sw(c)​ 差异很大

👉 会导致：

- 某些 channel 精度崩
- 动态范围浪费

所以：

> **per-channel multiplier + shift 是 NPU 精度的核心保障**
